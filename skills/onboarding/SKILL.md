# 사용자 온보딩 워크플로우

> Aha Moment를 정의하지 않고 온보딩 화면을 만들지 않는다.

## 게이트 (반드시 먼저)

온보딩 UI나 이메일을 설계하기 전에 반드시 완료해야 하는 선행 조건.

- [ ] Aha Moment가 정의되어 있는가 — "가입 후 N일 이내에 X 행동 완료 시 D30 리텐션 Y%"로 표현 가능해야 함
- [ ] 현재 Activation Rate 수치가 있는가 (가입자 중 Aha Moment 도달 비율, 7일 이내)
- [ ] 가입 → Aha Moment 현재 단계 수가 파악되어 있는가

## 규칙 (항상 따라야 함)

1. **Aha Moment 선행 정의**: 온보딩 플로우를 설계하기 전에 Aha Moment를 데이터 또는 가설로 먼저 명시한다. 정의 없이 단계를 설계하지 않는다
2. **Time-to-Value 5단계 이하**: 가입에서 Aha Moment 도달까지 단계를 5개 이하로 제한한다. 초과하면 제거할 단계를 먼저 찾는다
3. **각 단계 전환율 추적 의무화**: 온보딩 단계를 설계할 때 각 단계에 완료 이벤트를 정의한다. 측정 불가능한 단계는 설계하지 않는다
4. **스킵 옵션 필수**: 모든 투어/가이드/체크리스트에 "스킵" 또는 "나중에 보기"를 제공한다. 강제 진행은 절대 허용하지 않는다
5. **절대 빈 화면을 방치하지 마라**: Empty State에는 반드시 샘플 데이터 또는 행동 유도 CTA 1개를 포함한다
6. **절대 일률적 온보딩을 만들지 마라**: 사용자 역할/목표/숙련도 중 최소 1개 기준으로 경로를 분기한다
7. **이메일 시퀀스 행동 기반 분기**: Aha Moment에 이미 도달한 사용자에게 기초 안내 이메일을 보내지 않는다. 사용자 상태에 따라 이메일을 조건 분기한다

## 완료 체크리스트 (통과해야 완료)

- [ ] Aha Moment가 "행동 + 기간 + 리텐션 수치" 형식으로 정의되어 있는가
- [ ] 가입 → Aha Moment 단계가 5개 이하인가
- [ ] 각 온보딩 단계에 완료 추적 이벤트가 정의되어 있는가
- [ ] Activation Rate 측정 기준(분자/분모/기간)이 명시되어 있는가
- [ ] 모든 가이드/투어에 스킵 경로가 존재하는가
- [ ] Empty State에 CTA가 포함되어 있는가
- [ ] 사용자 세그먼트별 경로 분기가 설계되어 있는가
